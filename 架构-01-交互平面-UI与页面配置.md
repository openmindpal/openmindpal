# 细分架构设计：交互平面（UI）与页面配置

## 1. 目标与范围

本模块负责所有“面向用户的交互入口与展示”，但不承载任何旁路业务读写。所有数据获取与写入必须通过统一 API 请求链路（鉴权→校验→授权→执行→审计）。

覆盖范围：
- 基于 Schema 自动生成的通用业务 UI（列表/详情/表单/搜索/导出）
- 可定制的业务页面（少量关键域的定制交互）
- 可插拔的专业工作台（复杂交互：编辑器/画布/BI 等）
- 页面配置（租户/空间级）与个人偏好（用户级）
- 多端呈现策略（Web/移动/桌面/IM）

不在范围：
- 任何绕开 API 的直连数据源、脚本注入与自定义后端
- 业务规则的“前端固化”（规则应在 Schema/Policy/Tool/Workflow 表达）

## 2. 核心不变式与约束

- 唯一读写入口：UI 只调用 BFF/API；禁止直连数据库或内部模块
- 唯一鉴权源：身份与租户上下文由认证层建立；UI 不传入自造身份
- Effective Schema：前端只消费“按字段级权限裁剪后的 Schema 视图”
- 受控绑定：页面的数据绑定只能指向平台 API；动作绑定只能映射到 Tool/Workflow
- 版本化治理：租户/空间级页面配置进入发布流程、灰度与回滚；个人偏好不走发布

## 3. 信息架构与交互形态

### 3.1 生成式通用业务 UI（主路径）

输入：
- Effective Schema（实体/字段/关系/校验/展示配置）
- Policy 决策结果（导航可见性、字段可见/可写、动作可用性）

输出：
- 列表页：字段列、筛选器、排序、分页/游标
- 详情页：分组字段、只读/可写渲染、关联资源入口
- 表单页：字段校验、默认值、联动规则（规则以 Schema 表达）
- 搜索：结构化过滤 + 全文（可选） + 权限裁剪结果

关键点：
- 表单校验：前端基于 Schema 做即时提示，后端仍做强校验（防绕过）
- 字段级过滤：渲染层以 Effective Schema 为唯一来源，避免“拿到不可见字段元数据”

### 3.2 可定制业务页面（少量关键域）

原则：
- 允许更强的交互与信息密度，但数据读写仍然映射到平台 API 与受控动作
- 定制页面也必须只消费 Effective Schema；不可自行扩展字段含义

适用场景：
- 高价值域的工作流密集页面（审批、运营台、任务中心）
- 需要专业交互但不适合工作台插件化的页面（例如特定业务看板）

### 3.2.1 对话驱动导航（Mode A：OpenView + ViewParams）

目标：
- 用户通过对话表达意图后，UI 只打开“与当前意图相关的页面/面板”，避免把所有模块一次性展示

核心机制：
- 对话回执除文本外，还可携带 uiDirective，用于指导前端跳转或打开面板
- uiDirective 必须是“建议性指令”：前端必须校验可用性与权限，不能作为绕过治理的旁路

建议字段（最小集合）：
- openView：稳定的视图标识（如 calendar.week、tasks.today、notes.page）
- viewParams：视图参数（时间范围、过滤条件、默认选中资源等）
- openMode：page/panel/modal（可选；默认 page）

前端处理规则（建议）：
- 白名单校验：openView 必须存在于租户/空间已发布的 PageTemplate/Workbench 允许集合中，否则降级为默认页面或搜索页
- 权限一致：视图加载的数据仍通过 DataBinding→API 获取，并受 Effective Schema 与行级/字段级裁剪
- 动作受控：视图内任何写操作仍只能触发 ActionBinding→Tool/Workflow，并遵循风险分级与审批/确认

### 3.3 专业工作台（插件式）

定位：
- 对复杂交互形态提供独立“工作台插件”，例如编辑器、流程画布、BI 分析台

治理边界：
- 组件白名单：工作台使用平台允许的组件与依赖版本
- 数据源受控：数据访问通过 API；不得引入旁路通道
- 动作受控：所有写操作映射为 Tool/Workflow；声明风险与审批要求

#### 3.3.1 工作台插件最小标准（建议）

为让个人/团队/企业可以自行开发工作台插件且不破坏平台不变式，建议对“工作台插件”定义最小接入标准：

- 可注册：在 ComponentRegistry 中注册（name/version/entry/依赖约束），并可按租户/空间/环境启用或禁用
- 可配置：插件 props 具备结构化 Schema，支持页面配置引用与版本化回滚
- 可绑定：只通过 DataBinding 读取数据、只通过 ActionBinding 触发动作；绑定需声明 resource/action 并受权限裁剪
- 可治理：插件引用的工具与页面配置进入发布流程，支持灰度与回滚，并写审计
- 可观测：插件触发的 API/工具/作业具备 traceId 贯穿与基础指标（错误率、耗时、拒绝率）

#### 3.3.2 开发与接入流程（建议）

1) 开发插件：实现一个前端工作台模块（例如 editor/bi/canvas），仅调用平台 API，不直连外部数据源
2) 注册组件：将插件以组件形式注册到 ComponentRegistry，并声明版本与 props Schema
3) 绑定数据与动作：在 PageTemplate 中通过 DataBinding/ActionBinding 绑定到平台 API 与 Tool/Workflow
4) 治理发布：页面配置与组件版本走发布、灰度与回滚；高风险写动作按策略要求确认或审批
5) 运行期护栏：渲染层强制组件白名单与版本锁定；后端强制鉴权、授权、审计与内容治理

## 4. 页面配置契约（租户/空间级）

### 4.1 配置对象

- PageTemplate：页面模板（组件树/布局/数据绑定/动作映射），支持版本
- ComponentRegistry：组件注册表（白名单 + 版本），组件声明 props Schema 与安全边界
- DataBinding：受控数据绑定（只允许调用平台 API；绑定声明 resource/action 并受字段裁剪）
- ActionBinding：受控动作映射（只允许映射到 Tool/Workflow；声明风险等级与审批要求）
- LayoutVariants：同一页面配置的 mobile/desktop 变体

### 4.2 发布与治理流程

页面配置变更视为治理变更：
- 入口：仅具备治理权限的主体可提交与发布
- 风险门槛：涉及写动作的配置默认需要审查或双人复核；支持灰度与回滚
- 运行时防护：渲染层强制执行组件白名单、数据源限制、动作映射限制与版本锁定
- 审计：提交、审核、发布、回滚都必须落审计

与备份/恢复的关系：
- PageTemplate/ComponentRegistry/DataBinding/ActionBinding 属于配置数据，纳入备份与恢复范围
- 组件库实现与前端构建产物属于发布制品，通过版本化发布/回滚保障，不作为租户级备份内容

## 5. 个人级界面偏好（用户级）

### 5.1 目标

- 用户可自定义布局与展示偏好，仅影响自己
- 支持按端保存（desktop/mobile），可跨端同步与一键重置
- 不改变权限与数据范围；偏好只影响展示与交互

### 5.2 建议数据契约

- UserViewConfig：{ userId, scope(tenant/space), target(page/entity/viewId), variant(desktop/mobile), layout, visibleFields, sort, filters, shortcuts, version }
- PersonalDashboard：个人首页/快捷入口集合（由若干 target + shortcut 组合）

约束：
- DataBinding 与 ActionBinding 仍受控，且受字段级裁剪与权限约束
- 偏好保存默认低风险写入，可做轻量审计与配额/限流

与备份/恢复的关系：
- UserViewConfig/PersonalDashboard 属于用户数据，是否纳入备份取决于租户策略与合规要求
- 恢复建议默认落到隔离目标（新 space 或隔离环境），避免覆盖现网并支持比对

## 6. 多端入口策略

原则：
- 多端入口一致：Web/移动/桌面/IM 都属于 Channel Plane，只负责交互与展示
- 能力复用：复用同一 Schema/Policy/Tool/Workflow/Audit，不复制规则

落地优先级：
- 响应式 Web：覆盖大多数通用 CRUD；同一页面支持 mobile/desktop 变体
- 独立客户端壳（可选）：需要系统能力时采用，但仍只调用平台 API
- 工作台分级：桌面完整能力；手机端轻量模式（查看 + 轻编辑 + 快捷动作 + 审批/回执）

## 7. 关键交互流程（概要）

### 7.1 渲染流程（通用页面）

1) 获取导航与页面定义：按租户/空间与用户策略返回可见页面
2) 获取 Effective Schema：字段级裁剪后的实体 Schema 视图
3) 获取数据：使用 API 查询，服务端强制行级/字段级过滤
4) 提交动作：写操作映射到 Tool/Workflow，携带幂等键并进入审计

### 7.2 页面配置发布流程（租户/空间级）

1) 提交变更：PageTemplate/Registry/Bindings 作为变更集
2) 风险评估：是否含写动作、是否引用高风险工具
3) 审核与灰度：按空间/人群灰度启用
4) 回滚：一键回到上个稳定版本
5) 审计：全过程落库并可追踪

## 8. 可观测性与质量保障

- 指标：页面渲染耗时、API 调用耗时、错误率、权限拒绝率、工具触发成功率
- 追踪：前端 traceId 贯穿到 API 与后端链路
- 质量：Schema 驱动 UI 需要契约回归集（字段增删改对 UI 兼容性）

## 9. 演进路线

- MVP：通用 CRUD 生成 UI + Effective Schema 消费 + 个人偏好基础存储
- V2：页面配置版本化发布 + 写动作风险门槛 + 专业工作台插件化
- V3：多端入口与通知回执闭环 + 更强的页面治理与评测准入

## 10. 轻量美观 UI 规范（v1）

目标：
- 功能完整前提下，界面保持低噪声、低压迫、高一致性
- 让“治理/运维类页面”也具备现代产品的轻盈感

### 10.1 全局 Token（建议）

布局与尺寸：
- grid：12 列；contentMaxWidth=1200；gutter=24；pagePadding=24
- headerHeight=56；leftNavWidth=240（可折叠为 72）
- panelWidth=420（详情抽屉）；modalMaxWidth=720

间距（spacing）：
- xs=4，sm=8，md=12，lg=16，xl=24，2xl=32

圆角（radius）：
- sm=8，md=12，lg=16

边框与分割线（border）：
- hairline=1px
- borderColor=rgba(15, 23, 42, 0.10)
- dividerColor=rgba(15, 23, 42, 0.08)

阴影（shadow）：
- card: 0 1px 2px rgba(15, 23, 42, 0.06)
- float: 0 12px 32px rgba(15, 23, 42, 0.12)

字体（typography）：
- title: 18/28，weight 600
- body: 14/22，weight 400
- meta: 12/18，weight 400
- mono: 12/18（仅用于 traceId、runId、版本号等）

颜色（semantic）：
- bg: #ffffff；surface: #ffffff；surface2: rgba(15, 23, 42, 0.03)
- text: rgba(15, 23, 42, 0.92)；textMuted: rgba(15, 23, 42, 0.60)
- primary: #2563eb
- success: #16a34a；warning: #f59e0b；danger: #dc2626；info: #0ea5e9

状态徽标（badge）：
- 默认用“圆点 + 文本”表达状态
- 仅在危险/阻断态使用更强对比度；避免大面积背景色块

密度（density）：
- comfortable：rowHeight=44；tablePaddingX=12；tablePaddingY=10
- compact：rowHeight=36；tablePaddingX=10；tablePaddingY=6

### 10.2 页面信息层级（统一骨架）

所有页面统一两段式：
- PageHeader：面包屑/标题 + 1 行关键状态 + 1 个主按钮 + 1 个更多菜单
- PageBody：只呈现当前任务必需的信息，其余进入 Tabs/Accordion/Drawer

交互统一三件套：
- Drawer：详情/参数/证据/历史（保持上下文不丢失）
- Tabs：同一对象的不同视角（字段/关系/历史、步骤/日志/产物等）
- CommandPalette：全局搜索与跳转、常用动作入口（Ctrl/⌘+K）

## 11. 四类 PageTemplate（v1）

为避免页面风格漂移，建议仅提供 4 种高质量模板，并通过 PageTemplate 选择：

### 11.1 CRUD（通用生成页）

templateId：ui.crud.v1
- 适用：结构化实体的列表/详情/表单/搜索
- 版式：
  - 列表页：顶部轻量筛选条（chips）+ 表格/卡片列表 + 右侧详情 Drawer
  - 详情页：Summary + Tabs（Details/Relations/History）+ 右侧操作 Drawer
- 元数据来源：Effective Schema.uiHints 决定字段分组、主次字段、控件类型、列表主列

### 11.2 LOG（全屏日志表）

templateId：ui.logTable.v1
- 适用：审计流、运行记录、摄取作业等“长列表”
- 版式：
  - 默认 compact 密度
  - 行点击打开 Drawer 展示详情与折叠面板
- 强约束：主表只放 5–7 列；其余放 Drawer

### 11.3 MASTER_DETAIL（主从双栏）

templateId：ui.masterDetail.v1
- 适用：角色/成员、审批队列、知识集合列表等
- 版式：
  - 左：可搜索列表（支持键盘上下切换）
  - 右：详情（支持 Tabs）+ 底部固定操作条（如审批通过/驳回）

### 11.4 STUDIO/WIZARD（编辑/向导）

templateId：ui.studio.v1
- 适用：Bootstrap、智能体编辑、策略编辑（结构化编辑器）
- 版式：
  - 左：步骤/导航
  - 中：表单主体（每屏只处理一个问题）
  - 右：实时校验与风险提示（摘要 + 展开详情）

## 12. Schema 驱动的“美观”元数据（uiHints v1）

建议将 UI 相关信息收敛进 Schema.uiHints，并保持小而稳定：

字段级（Field.uiHints）建议：
- label、helpText
- control：text/textarea/number/select/date/datetime/switch/tags/json
- display：primary/secondary/meta
- group：summary/details/advanced 或自定义分组名
- list：{ show: true, order: 10, width: 240 }
- detail：{ show: true, order: 10 }

实体级（Entity.uiHints）建议：
- titleField：用于列表与详情主标题
- subtitleFields：用于列表第二行信息
- defaultSort：{ field, direction }
- defaultFilters：[{ field, op, value }]
- sections：[{ id, title, fields, collapsible, defaultCollapsed }]

示例（概念）：
```json
{
  "uiHints": {
    "titleField": "name",
    "subtitleFields": ["description", "updatedAt"],
    "sections": [
      { "id": "summary", "title": "概览", "fields": ["name", "status", "ownerId"] },
      { "id": "details", "title": "详情", "fields": ["description", "tags"], "collapsible": true }
    ]
  }
}
```

## 13. PageTemplate 与绑定的“轻量落地”约束（v1）

目标：
- 页面可以配置化生成，但视觉与交互保持一致
- 复杂信息通过 Drawer/Tabs 分层承载，不让页面变“厚”

约束建议：
- 每个模板只允许引用有限的组件集合（ComponentRegistry 白名单）
- 页面主操作最多 2 个；其余收纳到 More 菜单
- 长列表页面必须默认 compact；并提供一键切换为 comfortable
- 详情信息默认摘要化：input/output、policy 命中、trace 等以“摘要 + 展开”呈现

PageTemplate 关键字段（概念）：
```json
{
  "templateId": "ui.logTable.v1",
  "title": "审计",
  "bindings": {
    "data": [{ "id": "auditFeed", "type": "api", "ref": "audit.events.list" }],
    "actions": [{ "id": "export", "type": "tool", "ref": "audit.export" }]
  },
  "density": "compact"
}
```
