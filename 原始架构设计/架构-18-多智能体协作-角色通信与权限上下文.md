# 细分架构设计：多智能体协作（角色、通信、任务分配、权限上下文）

## 1. 目标与范围

本规范定义平台在支持多任务与长期运行时的多智能体协作方式：角色分工、通信协议、任务分配规则、权限上下文共享边界，以及与 Workflow/Job、审计与可观测的对齐。目标是在提升吞吐与可靠性的同时，避免越权、重复副作用与不可追溯。

## 2. 核心不变式

- 单一护栏：无论多少代理参与，所有读写都必须经过统一请求链路（鉴权→校验→授权→执行→审计）
- 上下文最小化：跨代理只共享必要摘要与引用（snapshotRef/digest），不共享敏感原文
- 写入单主：同一 run 对同一资源的写执行必须有唯一“写执行者”，避免并发副作用
- 责任可追溯：每个决策/工具调用都可定位到 agentRole、runId/stepId、traceId 与 policySnapshotRef

## 3. 角色模型（建议最小集合）

- Coordinator：会话入口与总体编排，拆解任务、分配子任务、汇总结果并对用户回执
- Planner：生成计划与依赖图（steps/guards），产出可执行的任务列表与风险提示
- Retriever：检索知识/记忆/资料，输出证据摘要与引用（evidenceRefs）
- Executor：执行工具调用与工作流触发，负责幂等键与执行状态推进
- Guard（Safety/AuthZ Reviewer）：在高风险动作前复核权限/外发/敏感策略，给出 allow/deny 与理由摘要
- Observer：长期监控与订阅处理（周期性检查、事件触发、告警与回执）

可选扩展：
- Debugger：对失败链路做根因归类与修复建议（不直接扩大权限）

## 4. 任务与执行单元

对象对齐：
- Task：对用户可见的目标（可能跨多个 Run/Job）
- Run：一次执行上下文快照（可由对话/触发器/审批通过创建或复用）
- Step：Run 内的最小执行单元（一次工具调用或子动作）
- Job：异步作业实例（导入/导出/备份/恢复/长任务）

约束：
- 一个 Step 对应一个 toolRef（或明确声明的子动作），并携带 policySnapshotRef 与 idempotencyKey
- 任意可重试 Step 必须幂等：重复执行不产生重复副作用

## 5. 代理间通信协议（概念级）

### 5.1 Agent Message Envelope

- from：{ agentId, role }
- to：{ agentId?, role? }
- correlation：{ taskId, runId?, stepId?, jobId?, requestId, traceId }
- intent：plan | retrieve | execute | review | observe | respond
- inputs：
  - userGoalDigest
  - constraints：{ allowedTools?, maxSteps?, maxCost?, timeBudgetMs? }
  - policySnapshotRef?
  - evidenceRefs?
  - contextRefs：{ memoryRefs?, schemaRefs?, connectorRefs? }
- outputs：
  - plan?: { steps[], guards[], expectedArtifacts? }
  - toolCall?: { toolRef, inputDigest, idempotencyKey, expectedOutputSchemaRef? }
  - decision?: { allow/deny, reasonSummary, matchedRulesDigest }
  - observation?: { outputDigest, errorCategory?, latencyMs? }

### 5.2 共享原则

- 只共享引用与摘要：input/output 以 digest + ref 形式传递
- 需要原文时：由接收方在自身上下文中重新走检索/授权获取，不允许代理间直接传原文绕过裁剪

## 6. 任务分配与并发控制

写入单主规则（建议）：
- 对同一 resourceRef 的写操作在同一时刻只允许一个 Executor 持有“写租约（write lease）”
- Coordinator 负责分配写租约，并在 step 完成或超时后释放

并发策略：
- 可并行：检索、总结、只读查询、不同资源的独立写入
- 必须串行：同一资源上的写操作、需要顺序一致性的步骤链

冲突处置：
- 检测到并发写冲突时，优先以乐观锁失败返回，并由 Planner/Coordinator 重新规划或提示用户

## 7. 权限上下文共享（Policy Snapshot 规则）

- Planner/Executor/Guard 在关键节点必须产出或引用 policySnapshotRef
- policySnapshotRef 只能通过 AuthZ 生成，代理不得自行构造
- 高风险动作（外发、批量、权限变更）默认要求 Guard 给出明确 allow/deny 与理由摘要

## 8. 长期运行与监控（Observer）

Observer 的职责：
- 订阅事件或定时触发检查，创建新的 Run/Job 并生成回执
- 对“阈值事件”（失败率、成本异常、权限拒绝异常、队列积压）触发告警与降级

约束：
- 监控触发的执行同样需要 idempotencyKey，避免重复告警/重复执行
- 长期任务必须可取消、可恢复，并能回放关键决策轨迹

## 9. 审计与可观测性对齐

审计要求：
- 每次代理决策与工具调用记录 agentRole、correlation（taskId/runId/stepId/jobId）、policySnapshotRef 与摘要
- 重要状态迁移（run/step/job）与回执发送都必须可追溯

可观测性：
- 指标：每角色调用量、成功率、拒绝率、平均耗时、重试次数、成本分布
- 追踪：traceId 串联 Coordinator→各代理→工具/工作流→外部系统

## 10. 演进路线

- MVP：Coordinator + Executor（单代理实现多角色也可）+ 写入单主 + 基础审计对齐
- V2：引入 Guard 与 Retriever，形成“计划-检索-执行-复核”闭环
- V3：引入 Observer 长期监控与多任务调度，多代理并行与容量治理完善
