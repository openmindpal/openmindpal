# 细分架构设计：离线同步（Sync：变更日志、冲突与可回放）

## 1. 目标与范围

同步模块支撑离线可编辑与多端一致性，通过“变更日志（Change Log）+ 状态投影（Projection）”模型实现可控的最终一致，确保冲突可解释、可修复，且同步过程可回放、可审计，并复用平台的权限裁剪与生命周期治理。

覆盖范围：
- 端侧变更日志：只追加的操作事件/补丁
- 服务端全局变更日志流与投影生成
- 增量同步：按游标/水位 push/pull
- 冲突检测与合并策略（按域/按字段配置）
- 幂等去重：OpId 去重与确定性摘要
- 同步审计：push/pull 作为受控 Tool/Workflow 执行

## 2. 核心不变式

- 端侧只追加：端侧操作以事件/补丁追加到本地日志
- 同步只传增量：按游标/水位拉取与推送，避免全量对账
- 权限一致：推送与拉取两侧都执行租户隔离与行级/字段级裁剪
- 可回放可审计：同步与合并过程可追溯，支持重放验证

## 3. 关键标识与最小数据集合

建议最小标识：
- ClientId：安装级客户端标识
- DeviceId：设备标识
- SpaceId：空间边界
- OpId：全局唯一操作号（ClientId + 单调序列或 UUID）
- Clock：单调时序标记（HLC/Lamport），用于排序与冲突解释
- BaseVersion：操作基于的投影版本（用于并发检测与冲突提示）

## 4. 操作表达

两类表达可二选一或组合：
- Patch：对结构化记录的 JSON Patch / 字段级变更（适配通用实体 CRUD）
- Domain Op：对特定域定义的高层操作（如 BlockEditor：insertBlock/moveBlock/replaceText）

建议所有操作都携带：
- schemaRef（实体/版本）
- targetRef（记录/对象标识）
- payload（补丁或 domain op 参数）
- contentDigest（用于审计与去重）

## 5. 冲突策略（按域/按字段配置）

默认策略（可配置）：
- 标量字段：LWW（Last-Write-Wins）并记录冲突轨迹（谁覆盖了谁）
- 集合字段：add/remove 的集合操作，服务端幂等合并
- 文本/富文本：协作编辑域优先 CRDT/OT（如 Yjs），同步层承载增量流
- 结构树（Block 树）：位置标识 + 可交换操作序列；必要时落到可视化冲突修复

冲突输出建议：
- conflicts[]：{ opId, targetRef, fieldPath?, reason, candidatesSummary, resolutionHints }

## 6. 幂等去重与确定性摘要

服务端要求：
- 对 OpId 做严格幂等：重复上报只返回已接收水位与合并结果摘要
- 合并结果生成 Deterministic Digest：同一输入序列得到同一摘要，用于回放与回归

## 7. 同步接口（契约级形态）

建议接口：
- sync.pull(spaceId, cursor, limit) -> { ops[], nextCursor, snapshotVersion }
- sync.push(spaceId, ops[], clientWatermark) -> { accepted[], rejected[], serverWatermark, conflicts[] }

安全要求：
- push/pull 都必须经过鉴权→校验→授权→审计链路
- 端侧缓存只保存 Effective Schema 与已授权数据的裁剪视图
- 对敏感域可启用端侧加密存储（以 Space/身份为密钥域）

## 8. 与工作流与审计的对齐

- 同步执行可视为受控 Tool/Workflow：
  - 大批量 push/pull 进入异步任务，支持重试/死信/限流
- 每次 push/pull 产生审计事件：
  - 输入摘要（opCount、范围、游标）
  - 输出摘要（accepted/rejected/conflicts 数量、nextCursor）

## 9. 可观测性

- 指标：同步延迟、冲突率、重试次数、幂等命中率、端侧积压
- 告警：冲突率异常升高、同步失败堆积、审计失败
- 追踪：opId 与 traceId/runId 关联，支持端到端排障

## 10. 演进路线

- MVP：增量 push/pull + OpId 幂等 + 基础冲突输出 + 审计
- V2：按域配置冲突策略 + 投影物化与回放摘要
- V3：端侧加密与隐私分区 + 协作编辑 CRDT/OT 深度集成
