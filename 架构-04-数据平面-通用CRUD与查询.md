# 细分架构设计：数据平面（通用 CRUD、查询、导入导出、备份恢复）

## 1. 目标与范围

数据平面负责“按 Schema 执行业务数据的通用读写”，并强制执行字段校验与权限过滤（行级/字段级）。数据平面不承接任何绕开 API 的直连访问。

覆盖范围：
- 按 Schema 的通用 CRUD 与查询
- 字段校验（类型/必填/约束）
- 行级约束与字段级裁剪（读与写）
- 导入导出（作为高风险能力时与 Workflow 联动）
- 备份与恢复（租户/空间隔离、加密、可审计）
- 查询优化：投影列/索引/物化（由 Schema 配置驱动）

## 2. 数据建模策略

推荐 Postgres + JSONB：
- 元数据（Schema/Policy/Workflow/Tool）走强结构表
- 业务数据可选两种模式并可演进：
  - 统一表 + JSONB payload：冷启动与快速扩展优先
  - 按域拆表 + JSONB 扩展字段：热点域逐步演进以提升性能与约束表达

建议能力：
- projectionColumns：把高频字段从 JSONB 投影到生成列/投影列
- indexes：对投影列、常用过滤条件、全文检索字段建立索引
- materializedViews：对复杂报表与聚合提供物化视图与增量刷新任务

演进阈值（建议用来判断何时“从 JSONB 演进”）：
- 查询：某实体出现稳定高频过滤/排序字段且 P95 延迟不可接受时，优先为该字段引入投影列 + 索引
- 搜索：需要稳定全文/模糊检索且命中字段固定时，建立专用索引或异步索引（必要时拆到检索侧）
- 导出：频繁大批量导出导致线上查询抖动时，导出必须异步化并使用物化投影/快照读取
- 报表：跨多维聚合成为常态且影响 OLTP 时，引入汇总表/物化视图或分析型存储
- 权限裁剪：行级/字段级规则复杂导致查询拼装成本高时，考虑引入“授权投影/物化”与按空间的预计算策略

## 3. 核心职责与边界

数据平面必须保证：
- 任何返回给上游的数据都已执行行级过滤与字段级裁剪
- 任何写入都已执行 Schema 校验、字段级可写约束、幂等与并发控制策略
- 不可信输入不得直接拼接为查询（统一结构化查询表达）

数据平面不负责：
- 身份与租户上下文建立（AuthN）
- 权限决策计算（AuthZ），但必须消费决策结果并强制执行裁剪
- 审计写入（Audit），但应返回必要摘要供 API 层审计

## 4. 通用 CRUD 的执行流程（概要）

### 4.1 读路径

输入：
- schemaRef（实体 + 版本）
- query（结构化过滤/排序/分页/游标）
- policyDecision（行级过滤表达 + 字段可见规则）

执行：
- 校验 query 与 schema 的匹配性（字段存在、类型正确）
- 应用行级约束（rowFilters）并与用户过滤合并
- 选择字段集合（字段级白名单/黑名单）并生成安全查询
- 返回裁剪后的结果集与摘要（count/nextCursor 等）

### 4.2 写路径

输入：
- schemaRef
- payload（结构化写入对象）
- policyDecision（字段可写规则、行级可写约束）
- idempotencyKey（必需）

执行：
- Schema 校验（类型/必填/约束）
- 字段级写入约束（只允许可写字段；禁止写入不可见/只读字段）
- 幂等去重（按幂等键 + 资源维度）
- 并发控制（乐观锁/版本号/条件更新策略，按实体配置）
- 返回结果与变更摘要（用于审计与回放）

幂等与并发落点（建议）：
- 幂等去重必须有持久化记录（至少记录 idempotencyKey、resourceRef、inputDigest、resultDigest、createdAt）
- 并发控制建议统一采用乐观锁（version/revision），并把冲突结果以可解释错误返回

## 5. 导入导出与批处理

原则：
- 导入导出属于高风险能力：涉及批量读写、越权与外泄风险
- 默认通过 Workflow/Automation 以异步任务执行（重试、死信、可取消）
- 导出数据必须字段级裁剪，且进入审计（谁在何时导出了什么口径的数据）

导入（建议约束）：
- 导入格式必须携带 schemaRef（实体 + 版本），并支持字段映射与默认值策略
- 导入必须先做校验与预检（dry-run）：返回将写入的记录数、会被拒绝的字段/行原因摘要
- 导入写入必须幂等：按导入批次 idempotencyKey 或 recordDigest 去重，避免重复写入
- 批量导入默认进入异步任务：支持限速、暂停/取消与失败续跑

导出（建议约束）：
- 支持按作用域导出（tenant/org/space/entity/query）并绑定权限快照，避免导出期间权限变更导致不可解释
- 支持流式/分片导出，输出为可治理的导出产物（artifact），下载受短期令牌控制
- 导出产物默认脱敏与最小化：字段级裁剪 + Safety/DLP 脱敏策略

## 6. 备份与恢复

目标：
- 在不破坏平台不变式的前提下提供可恢复能力，覆盖误操作、数据损坏与灾难恢复
- 将备份/恢复纳入治理：严格权限、审批门槛、可追溯审计与可回放作业

### 6.1 备份范围与隔离边界

- 逻辑范围：按 tenant/org/space 的隔离边界做备份与恢复，默认不允许跨租户恢复
- 数据范围：业务数据（Data Plane）、元数据（Schema/Policy/Tool/Workflow）、关键配置（页面配置、连接器元数据、用户偏好）按需分层备份
- 审计域：审计日志独立生命周期，默认单独备份/归档策略，不与业务数据混同回滚

覆盖范围拆分（建议）：
- 平台“功能数据”（应备份）：业务实体数据、Schema/Policy/Tool/Workflow 定义与版本、页面配置（PageTemplate/ComponentRegistry/Bindings）、个人偏好（UserViewConfig/PersonalDashboard）、连接器元数据（ConnectorType/Instance 与启用策略）
- 凭证数据（可选备份）：SecretRecord 的加密密文可随租户备份，但恢复必须满足密钥层级可用与审批门槛；默认不提供跨租户/跨环境的明文迁移能力
- 运行态数据（不备份）：缓存、队列临时状态、短期会话等可再生数据

不覆盖项（边界声明）：
- 前端代码与组件库实现本身、后端服务代码、运行时镜像等属于“发布制品”，由制品仓库与发布治理保证可回滚，不归入租户级数据备份
- 外部系统的源数据不在平台内存储的部分不在备份范围内（只备份连接器配置与同步/导入产物）

### 6.2 备份类型（建议组合）

- 逻辑备份（Logical Backup）：按 Schema 导出结构化数据与元信息，便于租户级迁移与精细恢复
- 存储快照（Snapshot）：用于快速恢复整库或大范围数据（需与治理与环境隔离对齐）
- 时间点恢复（PITR）：基于数据库 WAL/日志的时间点回滚能力（更偏灾备与事故恢复）

### 6.3 恢复策略（建议默认“恢复到新空间”）

- 默认恢复到新 space（或隔离环境）：避免覆盖现网数据，支持比对后再合并/切换
- 恢复作业必须可回放：固化 backupRef、policySnapshot、恢复目标与参数摘要
- 恢复前强制预检：版本兼容性（Schema 版本）、字段映射、冲突策略与影响面摘要
- 恢复后校验：记录数/校验失败数、关键实体一致性摘要，产出恢复报告并写审计

### 6.4 安全与合规

- 加密：备份产物静态加密，密钥分区与 Secrets/Key Contract 对齐
- 访问控制：备份创建/下载/恢复均需显式权限与高风险审批门槛
- 留存：按 tenant 配置 retention 与 legal hold；删除备份也必须审计

## 7. 分析与报表（BI/OLAP）集成

目标：
- 不以 OLAP 能力阻塞 MVP，但要保证可演进集成且权限一致

建议路径：
- 早期：基于 Postgres 物化视图/汇总表 + 增量刷新任务
- 进阶：引入分析型存储（列存/数仓）
- 同步：批处理 ETL/ELT 或 CDC 增量同步
- 权限一致：分析侧复用租户隔离与行级/字段级授权（至少租户/组织约束）
- 可追溯：报表查询与导出同样进入审计

## 8. 安全与治理

- 多租户隔离：所有查询必须绑定 tenant/space 作用域
- 权限强制：数据面不能信任上游“已过滤”，必须基于决策结果做最终裁剪
- 数据最小化：默认不返回敏感字段；脱敏策略与 Safety/DLP 对齐
- 配额与限流：对热点实体、批量操作设置租户级与用户级阈值

## 9. 可观测性

- 指标：查询延迟、慢查询、缓存命中率、导入导出成功率、幂等命中率
- 追踪：为每次 CRUD/批处理产生可关联的 traceId 与 run/step 摘要
- 审计对齐：输出必要摘要字段供 Audit 写入（不包含敏感原文）

## 10. 演进路线

- MVP：通用 CRUD + Schema 校验 + 资源级权限约束 + 审计摘要
- V2：行级/字段级强裁剪 + 导入导出异步化 + 备份恢复作业 + 投影列/索引能力
- V3：物化与分析集成 + 更强的查询表达与回放评测
