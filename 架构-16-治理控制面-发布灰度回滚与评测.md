# 细分架构设计：治理控制面（Governance / Admin）

## 1. 目标与范围

治理控制面负责平台可治理性：组织/空间管理、策略与配置变更治理、审计导出，以及 Schema/Policy/Workflow/Skill/页面配置等核心契约的发布、灰度与回滚，并把“回放评测”作为变更准入门槛的一部分。

覆盖范围：
- 组织/空间（Tenant/Org/Space）管理与成员体系
- 治理权限：发布/审批/回滚/导出等治理动作授权
- 变更治理：Schema/Policy/Workflow/Tool/Skill/PageTemplate 的版本化发布
- 灰度与回滚：按租户/空间/环境启用与回退
- 审计导出：对接合规与 SIEM（可选）
- 数据治理作业：导入/导出、备份/恢复的权限、审批、留存与追溯
- 用户私有自举 Skill：草稿生成、私有启用、回滚禁用、评测与审计闭环
- 评测准入：兼容性检查、风险检查与回放评测绑定

## 2. 核心不变式

- 治理变更必须可追溯：提交/审核/发布/回滚都进入审计
- 默认拒绝：新能力（工具/连接器/策略）默认不可用，需治理启用
- 契约优先：变更以 Schema/Policy/Workflow/Tool 等契约为边界，禁止旁路发布
- 可回滚：任何上线变更都必须有回滚路径与上一稳定版本

## 3. 治理对象与生命周期

建议治理对象：
- Schema（实体/字段/校验/展示）
- Policy（授权策略与解释器配置）
- Tool/Skill（能力单元与风险元数据）
- Workflow（审批与编排）
- PageTemplate（页面配置与动作绑定）
- Connector（连接器启用、scope 变化与外发策略）
- Data Job（导入/导出/备份/恢复作业与产物生命周期）

生命周期（建议）：
- draft → reviewed → released → deprecated → retired

用户私有 Skill 生命周期（建议）：
- draft → enabled_user_scope → disabled/revoked

## 4. 发布流程（概念级）

通用发布流程：
1) 提交变更（变更集 + 目标范围：tenant/space/environment）
2) 自动检查：
   - 兼容性检查（Schema/Tool/Workflow/Policy 契约）
   - 风险检查（写动作、外发、权限扩大、批量影响）
3) 审核与审批（按风险等级决定单人/多人/分级）
4) 灰度发布（按空间/人群/环境逐步启用）
5) 监控与回归（关键指标与回放评测）
6) 回滚（快速切回上一稳定版本）
7) 审计记录（全过程）

用户私有自举 Skill（建议流程）：
1) 生成草稿：AI 仅允许创建 draft（包含 tool contract 与风险声明），默认不可执行或仅 dry-run
2) 自动检查：Schema 校验、依赖与出站声明检查、风险分级校验
3) 用户确认或审批：涉及外发、连接器凭证、批量写入的默认需要确认或审批
4) 私有启用：仅对 user 或 user+space 生效，绑定配额/限流与出站白名单
5) 回滚与禁用：随时撤销启用，撤销动作写审计

## 5. 评测与准入（Release/Eval Contract）

目标：
- 把“可回放评测”作为变更门槛，降低回归风险

评测范围示例：
- 工具正确性：参数生成/输出结构是否符合 Schema
- 安全回归：权限裁剪、注入防护、敏感信息过滤
- 行为指标：成功率、拒绝率、人工介入率、回滚率

准入原则：
- 不通过评测不发布
- 发布后支持快速回滚并写审计

### 5.1 准入门槛分层（建议）

为避免治理成为生态扩展吞吐的瓶颈，建议把门槛做成分层可配置，并尽量自动化。

强制门槛（所有变更）：
- 契约校验：Schema/Tool/Workflow/Policy 格式与版本语义校验通过
- 兼容性检查：对外接口与存量数据的兼容性结论明确（兼容/非兼容+迁移计划）
- 风险评估：写入影响面、外发、权限扩大、批量影响、敏感数据触达范围
- 回滚可用：上一稳定版本可一键切回，且不依赖手工修复

建议门槛（高风险变更）：
- 回放评测集：覆盖成功路径、拒绝路径、失败分类与降级路径
- 差异预检：提供 dry-run 产物（写入差异、外发摘要、影响面统计）
- 灰度策略：按空间/人群/环境灰度，设定自动回滚阈值

### 5.2 回放评测的最小形态（建议）

回放评测应以“轨迹与摘要可比”为核心，而不是依赖模型输出可复现。

建议至少具备：
- 测试用例：{ inputs, expectedToolCalls/constraints, expectedDecisionSummary, expectedOutputDigestConstraints }
- 判定维度：工具选择正确性、授权决策一致性、字段裁剪一致性、审计完整性、外发策略一致性
- 门槛策略：按风险等级配置通过阈值（例如拒绝率波动上限、误操作率上限）

自举 Skill 的准入门槛（建议）：
- 最低：契约校验通过（input/output schema、riskLevel、idempotency、timeouts）
- 涉及外发/写入：必须通过 dry-run 预检与风险门槛（确认/审批）
- 可选：对关键能力引入最小回放评测集（至少覆盖失败模式与拒绝路径）

## 6. 权限与分权（最小授权）

建议角色分离：
- Tenant Admin：租户级治理（空间管理、策略总开关）
- Space Admin：空间级治理（页面配置、工具启用、成员管理）
- Data Admin：数据治理（导入/导出、备份创建、恢复发起与审计导出）
- Auditor/Compliance：审计检索与导出

用户私有自举 Skill 的权限建议：
- 默认拒绝：是否允许“生成私有 skill draft”由租户/空间策略开关控制
- 最小权限：私有 skill 仅可访问 ownerScope 内资源与被授权的数据域
- 禁止升级：私有 skill 不允许自动发布到 space/tenant 级共享

委派与分权：
- 组织/空间委派管理与最小授权，避免把治理权限集中到单点管理员

## 7. 审计导出与保留策略

- 导出行为本身进入审计
- 支持 retention 与 legal hold（按合规要求）
- 可选对接 SIEM（结构化事件流）

## 8. 导入/导出、备份/恢复治理要点

- 权限门槛：导出、备份、恢复默认高风险；恢复与大范围导出建议强制多级审批
- 默认隔离恢复：恢复优先落到新 space 或隔离环境，避免直接覆盖现网
- 产物治理：导出/备份 artifact 必须加密、设置 retention，并通过短期令牌下载
- 删除可追溯：删除备份或导出产物同样进入审计，满足合规留存要求
- 配置覆盖：页面配置（PageTemplate/Registry/Bindings）默认纳入备份；个人偏好是否纳入由租户策略决定
- 凭证边界：SecretRecord 默认仅备份加密密文；恢复需满足密钥可用与审批门槛，不提供跨租户明文迁移

## 9. 可观测性

- 指标：发布频率、回滚率、兼容性失败率、评测失败率、治理操作错误率
- 告警：权限异常扩大、外发策略放宽、高风险工具被启用
- 追踪：变更集与运行期指标、审计事件与回放评测关联
- 变更吞吐：平均发布周期、人工审核占比、自动检查耗时、回滚平均耗时（用于识别治理是否成为扩展瓶颈）

## 10. 演进路线

- MVP：基础治理界面 + Schema/Policy/Tool/Workflow 发布与回滚 + 审计导出
- V2：灰度发布 + 自动兼容性/风险检查 + 评测准入
- V3：企业级合规联动（SIEM/Legal Hold）+ 更强的多环境分发与运营体系
