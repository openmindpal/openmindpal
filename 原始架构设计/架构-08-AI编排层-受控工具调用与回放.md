# 细分架构设计：AI 编排层（AI Orchestrator）

## 1. 目标与范围

AI 编排层将“用户意图”转化为“受控的工具/流程调用”，并把智能体行为收敛为可治理、可审计、可回放的执行轨迹。AI 本身不直接读写数据，只能通过已注册的 Tool/Skill 与 Workflow 执行动作。

覆盖范围：
- Tool/Skill 注册表与能力发现（名称/版本/能力/风险/启用范围）
- 工具参数与返回的 JSON Schema 校验
- 工具调用策略：风险分级、是否需确认/审批、速率与配额
- 运行上下文固化：Subject/Tenant、ToolRef、Policy Snapshot、输入输出摘要、幂等键
- 回放：按同一 toolRef + policySnapshot + inputDigest 复现可解释轨迹

不在范围：
- 模型路由与成本治理（由 Model Gateway 负责）
- 工具执行隔离（由 Skill Runtime 负责）

## 2. 核心不变式

- AI 只能调用注册的 Tool/Skill
- 工具调用必须走统一链路：鉴权→校验→授权→执行→审计
- 高风险写操作默认进入 Workflow（审批/双人复核）
- 扩展只走契约：Tool/Skill、Workflow、Policy、Schema

## 3. Tool/Skill 契约（关键字段）

Tool Contract（最小集合）：
- name、version
- scope（read/write）
- resourceType、action
- inputSchema、outputSchema
- riskLevel、approvalRequired
- rateLimit/quota、timeouts
- idempotency（required/strategy）

工具治理要求：
- 可发现：注册表可查询（能力、风险、启用租户/空间）
- 可验证：入参/出参 Schema 可自动校验与兼容检查
- 可授权：映射到 Resource/Action，纳入统一 AuthZ
- 可治理：默认拒绝，按租户/环境灰度启用；支持配额、限流、并发与熔断
- 可审计：每次执行产生可回放摘要（版本、依赖、耗时、错误分类）

## 4. 运行上下文固化（可回放对象）

每次工具执行建议固化为可序列化对象：
- subject/tenant/space
- toolRef（工具名称 + 版本 + 依赖摘要）
- inputDigest/outputDigest（按脱敏策略）
- policySnapshot（执行时刻权限快照引用或摘要）
- idempotencyKey（写操作必需；审批通过后复用）
- runId/stepId（与 Workflow 执行单元对齐）
- traceId（端到端追踪）

### 4.1 回放与重执行（区分两类动作）

为避免把“可回放”误解为“重新运行即可复现”，建议在编排层把动作严格分为两类：
- 回放（replay）：重建已发生过的决策轨迹与结果摘要，用于复盘、追责与回归评测；不再次调用模型与外部系统
- 重执行（re-exec）：以新的 runId 在当前环境重新执行（可能换模型、权限、外部数据已变化）；必须生成新的审计链路，并显式标记 re-exec

回放对象必须满足：
- 轨迹可解释：选择了哪个工具、为何允许/拒绝、命中哪些规则摘要
- 结果可校验：关键产物有结构化摘要与一致性校验（digest），可用于回归集对比
- 证据可追溯：涉及检索、外发与高风险写入时，保留最小必要证据并受保留期与脱敏策略约束

## 5. 工具调用编排流程（概要）

### 5.1 只读工具

1) 选择 toolRef（依据能力/风险/成本/上下文）
2) 参数生成并校验 inputSchema
3) 授权计算：资源/行/字段级
4) 执行工具（受控读写接口）
5) 校验 outputSchema + 脱敏裁剪
6) 写审计（含决策摘要与输入输出摘要）

### 5.1.1 对话回执与 UI 指令（OpenView + ViewParams）

为支持“对话驱动导航”的交互体验，编排层可在对话回执中附带 uiDirective，用于指导前端打开与当前意图最相关的页面/面板。

建议字段（最小集合）：
- openView：稳定视图标识（如 calendar.week、tasks.today）
- viewParams：视图参数（时间范围、过滤条件、默认选中资源等）
- openMode：page/panel/modal（可选）

约束：
- uiDirective 只能用于导航与展示，不改变任何权限与数据范围
- 前端必须对 openView 做白名单与权限校验，不可将其作为旁路执行入口
- 任何写操作仍必须通过 Tool/Workflow 触发，并进入审计与风险门槛

### 5.2 写工具（低风险直写）

在满足策略允许与风险门槛的情况下：
- 必须携带/生成 idempotencyKey
- 必须在执行前固定 policySnapshot
- 审计记录包含写入摘要与去重结果

### 5.3 写工具（高风险需审批/异步）

1) 提交写意图：toolRef + inputDigest + idempotencyKey
2) 授权计算并产出 policySnapshot
3) 创建 Workflow/Run（锁定 toolRef、policySnapshot、inputDigest）
4) 审批通过后执行 steps（可重试、可补偿）
5) 逐步写审计（进入流程、审批、每步执行、补偿/撤销）

## 6. 风险分级与护栏

风险来源：
- 写入影响面（批量/跨域/外发）
- 涉及凭证与外部系统调用
- 导出/分享/权限变更等高敏动作

护栏策略（建议）：
- 只建议/需确认/需审批/自动执行（可回滚）四档自治等级
- 默认拒绝：新工具、新连接器、新工作流需治理启用
- Dry-run：高风险动作支持预览差异与分步确认

## 7. 与 Knowledge/Memory 的协作

- 检索与记忆写入都必须映射为工具调用
- 记忆写入默认需要显式动作（用户确认或策略允许），并进入审计
- 检索证据链：检索摘要与引用片段进入审计，支持追溯

## 8. 可观测性与评估

- 指标：工具调用成功率、拒绝率、人工介入率、回滚率、平均步数与耗时
- 追踪：对话→计划→工具→工作流/队列→外部系统 全链路 trace
- 评测：变更（工具/提示模板/编排策略）应绑定回放评测集与准入门槛

## 9. 演进路线

- MVP：工具注册 + 参数/返回 Schema 校验 + 基础风险分级 + 审计
- V2：审批/回放/幂等贯通 + 多入口触发（定时/事件/订阅）
- V3：多代理协作（共享权限上下文）+ 更强的评估与运营体系
