# 细分架构设计：工作流与自动化（Workflow/Automation）

## 1. 目标与范围

工作流模块承载高风险写操作的“先审后改”、异步任务编排与可靠执行能力，为平台提供可重试、可回放、可撤销/可补偿的执行框架，并与审计、授权、幂等治理闭环。

覆盖范围：
- Workflow 定义：审批步骤、触发条件、执行计划（steps）
- 队列与任务：异步执行、重试、退避、死信
- 幂等去重：幂等键贯穿对话/审批/执行
- Run/Step：执行上下文快照与步骤单元
- 补偿与撤销：Undo/Compensating Tool/Saga 编排

## 2. 核心不变式

- 高风险写操作默认进入 Workflow（审批/双人复核）
- 执行前必须完成授权计算，并固化 Policy Snapshot
- 每次执行与重试都必须写审计
- 重试必须幂等：重复执行不产生重复副作用

回放与重执行约束：
- 回放（replay）用于复盘、追责与回归评测，默认重放已记录的 Run/Step 轨迹与摘要，不再次调用模型与外部系统
- 重执行（re-exec）以新的 Run 创建新的执行链路，允许在当前环境重新调用模型/外部系统，但必须显式标记并单独审计
- 幂等键语义不跨 re-exec 复用：re-exec 默认生成新的幂等键或新的幂等域，避免把历史执行与新执行混淆为同一副作用

## 3. Workflow 契约（最小字段）

Workflow Contract（最小集合）：
- name、version
- trigger（toolRef/事件/定时）
- approvalSteps
- executionPlan（steps）
- retryPolicy、deadLetterPolicy
- compensation（可选）
- serialization：policySnapshot、toolRef 锁定、inputDigest 绑定

执行单元建议：
- Step：{ toolRef, inputDigest, policySnapshot, idempotencyKey, timeout, retryPolicy }
- Run：一次执行的上下文快照（对话/触发/审批通过后复用同一 RunId）

## 3.1 执行对象与状态机（建议固化）

对象定义（概念级）：
- Job：一类异步作业的实例（导入/导出/备份/恢复），对外暴露进度与可取消能力
- Run：一次执行上下文快照（可由对话、触发器或审批通过后创建/复用）
- Step：Run 内的最小执行单元（通常对应一次工具调用或一次子动作）
- Artifact：导出/备份等产生的可下载产物（受治理与审计）

状态机（建议）：
- Job：queued → running → succeeded | failed | canceled
- Run：created → running → succeeded | failed | canceled | compensating | compensated
- Step：pending → running → succeeded | failed | skipped | timeout | compensating | compensated
- Artifact：created → ready → expired | deleted

幂等范围（建议）：
- 写意图幂等：同一（tenant/space + action + resourceRef + idempotencyKey）只能产生一次有效副作用
- Step 幂等：同一（runId + stepId + attempt）只记录一次状态迁移与审计事件
- Artifact 幂等：同一 jobId 的产物生成按输出摘要去重，重复生成返回同一 artifactRef

## 4. 审批模型

审批步骤设计要点：
- 按风险等级与资源影响面决定审批链路（单人确认/多人复核/分级审批）
- 审批通过后执行必须复用原始 inputDigest、policySnapshot 与 idempotencyKey
- 审批结果与链路摘要写入审计（包含拒绝原因）

## 5. 队列与重试策略

建议能力（以 BullMQ 起步的抽象语义）：
- retry：固定次数 + 退避（指数/抖动）
- dead letter：超过阈值进入死信并提供人工处置入口
- idempotency：按（idempotencyKey + step/toolRef）去重
- timeout：步骤超时终止并进入可恢复状态

任务状态：
- pending / running / succeeded / failed / canceled / compensating / compensated

## 6. 补偿、撤销与 Saga

撤销策略：
- Delay Window：对高风险操作提供延时执行窗口，窗口内允许取消
- Undo Token：可撤销操作由工具返回 undoToken 或最小补偿信息
- Compensating Tool：不可直接撤销的外部副作用必须声明补偿工具与语义

跨系统一致性：
- 使用编排式 Saga：每一步落审计，可从任意步骤恢复或进入补偿链路
- 失败分类：可重试/需人工/需审批/需降级，驱动后续处置

## 7. 与 Tool/Skill Runtime 的协作

工作流只编排受控工具：
- 工具必须声明 input/output Schema、风险等级、审批要求、幂等策略
- 工具执行仍受统一请求链路约束（鉴权→校验→授权→审计）
- 工具版本锁定：toolRef（名称 + 版本 + 依赖摘要）用于回放一致性

## 8. 典型作业：导入、导出、备份、恢复

导入作业（Import Job）建议：
- 先预检（dry-run）：Schema 校验、字段映射与行级/字段级拒绝原因摘要
- 再执行写入：分批提交，幂等去重，可限速、暂停与取消
- 失败续跑：记录已完成批次水位，支持从检查点恢复

导出作业（Export Job）建议：
- 固化 policySnapshot：避免导出期间权限变化导致不可解释结果
- 流式/分片产出 artifact：支持大数据量导出与断点续传
- 下载治理：短期令牌、次数限制与审计记录

备份作业（Backup Job）建议：
- 支持按 scope（tenant/space）触发与定时策略
- 备份产物加密与留存策略执行，删除/过期同样进入审计

恢复作业（Restore Job）建议：
- 默认恢复到隔离目标（新 space 或隔离环境），避免覆盖现网
- 恢复前强制预检：Schema 版本兼容、冲突策略与影响面摘要
- 恢复后产出报告：校验统计与异常摘要，供比对与复盘

## 9. 可观测性与运营

- 指标：审批耗时、队列堆积、任务成功率、重试次数、死信增长、撤销/补偿率
- 追踪：runId/stepId 串联到 traceId，便于端到端排障
- 告警：队列积压与死信增长、审计写入失败、授权异常波动

## 10. 演进路线

- MVP：基础 Workflow 定义 + 队列任务 + 重试/死信 + 审计
- V2：幂等贯穿 + Run/Step 快照 + 可取消与部分补偿
- V3：Saga 编排标准化 + 更强的风险分级与评测准入联动
