# 细分架构设计：元数据平面（Schema Registry）

## 1. 目标与范围

元数据平面是平台“大脑”，负责用可版本化的 Schema/Model 描述实体、字段、关系、校验与展示配置，并为其他模块提供稳定契约与兼容性规则。

覆盖范围：
- 实体/字段/关系/校验/展示配置（uiHints）定义与版本管理
- Schema 兼容规则、发布流程与回滚
- 迁移策略定义（惰性迁移/在线迁移/双写回填）
- 投影/索引/物化建议（为数据面优化提供元信息）
- Effective Schema 生成的输入侧（字段级可见/可写规则的表达）

不在范围：
- 实际业务数据读写（由数据平面负责）
- 权限决策计算（由 AuthZ 负责）

## 2. 核心概念与数据结构

### 2.1 Schema/Model（建议字段集合）

Schema Contract（最小集合）：
- name、version
- entities（实体集合）
- fields（type/required/validation）
- relations（关联）
- uiHints（展示配置）
- projections/indexes（投影与索引提示）
- fieldVisibility（字段级读/写可见规则，用于生成 Effective Schema）

### 2.2 版本语义与兼容性

建议至少区分：
- 兼容变更：新增可选字段、增加枚举值（视策略）、新增索引提示等
- 非兼容变更：字段类型变更、删除字段、将可选改必填等

推荐规则：
- 删除字段：先废弃（deprecate）后移除，提供兼容窗口
- 新增必填字段：必须具备默认值/回填策略，并绑定迁移计划
- 类型变更：视为非兼容，必须提供迁移与回滚预案

### 2.3 扩展命名空间与前向兼容（避免 Schema 无限膨胀）

为支持大量行业/团队差异而不破坏平台稳定性，Schema Contract 应将可变部分收敛为“命名空间化扩展”，并对扩展字段采取前向兼容策略。

建议规则：
- 稳定字段严格：released 版本只允许使用已定义的稳定字段集合，避免顶层字段随意增长
- 扩展字段命名空间化：可扩展内容统一放入 extensions（键为命名空间），例如 org.example.xxx
- 未知扩展可忽略但保留：解析器对未知 extensions 命名空间默认忽略其语义，但在存储与回滚中保持原样
- 禁止语义漂移：同一稳定字段与同一扩展键在同一主版本内不得改变语义；行为变化必须通过版本语义表达
- 扩展可注册：对关键扩展可引入扩展注册表（schemaExtensionType），用于校验与发布准入

典型可扩展点（建议放入 extensions）：
- 自定义字段类型的渲染/编辑器 hints
- 行业特定校验规则（在平台允许的校验表达式集合内）
- 领域特定投影/索引提示（由数据面支持的能力子集）

## 3. 模块职责边界

元数据平面输出：
- Schema 定义（按版本）
- 兼容性检查结果（用于发布准入）
- 迁移策略与作业定义（供 Workflow/Automation 执行）
- Effective Schema 生成所需的字段规则表达（与 AuthZ 联动）

元数据平面不直接做：
- 任意业务逻辑判断
- 对业务表的直接读写

## 4. 核心接口（契约级）

建议对外提供以下能力（被 API 层调用）：
- Schema 获取：按 name/version 获取 Schema
- Schema 列表与检索：按标签/域/状态（draft/released/deprecated）筛选
- Schema 发布：提交变更、兼容性检查、生成发布版本
- Schema 回滚：回到上一稳定版本（与治理流程绑定）
- Effective Schema：结合 AuthZ 的字段级规则，生成对某主体/空间的裁剪视图

## 5. Schema 演进与迁移机制

### 5.1 迁移策略

- 惰性迁移：读写时自动升级 payload（配合版本标记）
- 在线迁移：后台任务批量回填（限速、可恢复）
- 双写/回填：对关键字段短期双写以保证一致

### 5.2 迁移执行边界

- 迁移的实际执行由 Workflow/Automation 承载（队列任务、重试、死信）
- 元数据平面只负责定义迁移计划与兼容窗口，并提供必要的校验规则

## 6. 与数据平面的契合（JSONB 与投影）

推荐 Postgres + JSONB：
- 元数据走强结构表（便于检索与版本治理）
- 业务数据可采用统一表 + JSONB payload 或按域拆表 + JSONB 扩展字段

性能增强由元数据驱动：
- 对高频查询字段配置投影列/生成列/索引策略
- 对热点实体可逐步引入按域拆表或物化视图，仍由 Schema/迁移策略驱动演进

## 7. 变更治理与审计

- Schema 变更属于治理变更：提交、审核、发布、灰度、回滚都写审计
- 发布准入：必须通过兼容性检查与风险检查（字段级权限影响、索引影响、迁移影响）
- 版本锁定：API/工具/流程引用 Schema 时应携带 version（或解析到确定版本）

## 8. 可观测性与质量保障

- 指标：Schema 发布频率、兼容性失败率、迁移成功率/耗时、Effective Schema 生成耗时
- 回归集：用典型用例覆盖字段增删改、权限裁剪、UI 生成兼容性
- 追踪：Schema 发布与迁移 Run/Step 串联 traceId

## 9. 演进路线

- MVP：Schema Registry + 版本管理 + 基础兼容规则
- V2：迁移策略与作业联动 + Effective Schema 标准化生成
- V3：更强的语义层（指标口径/分析映射）与发布评测准入
