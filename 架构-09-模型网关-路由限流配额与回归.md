# 细分架构设计：模型网关（Model Gateway）

## 1. 目标与范围

模型网关为平台提供多模型接入、路由、降级与熔断，并在统一链路中实现速率限制、配额与成本归集（用于治理与预算控制）。模型网关不承担业务逻辑，只提供“可控的模型调用能力”。

覆盖范围：
- 多模型接入与统一调用接口
- 模型提供方适配（云端 API / 自建模型），以及凭证与出站治理对齐
- 路由策略：按场景/租户/任务类型选择模型
- 降级链路：快模型优先、强模型兜底；失败切换与重试策略
- 速率限制与配额：租户/用户/场景维度
- 成本归集：按租户/用户/工具/模型维度统计（用于治理，不依赖计费）
- 输出治理对齐：结构化输出约束与敏感信息过滤（与 Safety/DLP 协作）
- 版本回归：模型/提示模板变更绑定离线回放评估与可回滚

## 2. 核心不变式

- 模型调用也必须可审计：记录路由决策、输入输出摘要、耗时与错误分类
- 输出必须受治理：结构化约束、敏感信息过滤与外发策略对齐
- 配额与限流面向治理：不引入计费依赖，但必须可预算与限额

## 3. 统一模型调用契约（概要）

建议统一请求结构（概念级）：
- modelRequest：{ tenant/space, subject, purpose, constraints, inputDigest, outputSchema?, maxTokens?, timeout? }
- routingDecision：{ provider, model, fallbackChain, reason }
- modelResponse：{ output, outputDigest, safetyFlags, usage, latencyMs, traceId }

关键要求：
- outputSchema：工具参数与关键产物优先使用 JSON Schema 约束（减少返工与多轮重试）
- input/output digest：按脱敏策略记录摘要而非原文（与审计对齐）

## 4. 模型提供方接入与凭证托管

### 4.1 Provider Adapter（提供方适配）

模型网关通过“提供方适配层”对接多种模型来源：
- 云端 API：按 provider/model 路由（支持多提供方与多模型）
- 自建模型：以 provider=internal/selfhosted 的形式纳入同一契约（同样纳入限流、审计与回归）

建议抽象（概念级）：
- ProviderAdapter：{ providerId, supportedModels, invoke(request) }
- ModelCatalog：{ providerId, modelId, capabilities, defaultLimits, pricingHint? }

### 4.2 凭证与密钥（Secrets/Connector 对齐）

原则：
- 模型调用所需 API Key/Token 不进入提示词、日志与审计原文，仅以脱敏摘要记录
- 凭证按 ownerScope（tenant/space/user）隔离，并支持轮换/撤销
- 不同租户可配置不同提供方与不同凭证（BYOK），并可按环境分离

使用路径：
- modelRequest 触发路由后，按（tenant/space + providerId）解析到可用的 connectorRef/secretRef
- 运行期仅在调用前受控取用凭证，调用完成后不在内存长期驻留

出站治理：
- 提供方域名/路径纳入 EgressPolicy 白名单（可选强制代理）
- 与 Safety/DLP 联动：对输入输出摘要做脱敏与风险标记

## 5. 路由、降级与熔断

### 4.1 路由维度

- 场景：对话规划/工具参数生成/总结/检索重排等
- 租户：企业/个人、合规要求、预算等级
- 任务：时延敏感/质量敏感/结构化输出

### 4.2 降级策略（从快到准）

- 快模型优先，强模型兜底
- 失败切换：同提供方不同模型，或跨提供方
- 上下文治理：压缩/摘要/裁剪后重试，控制 token 上限

### 4.3 熔断与重试

- 熔断：按模型与提供方维度统计错误率/超时，触发熔断与切换
- 重试：只对可重试错误重试，并记录 attempt 与退避策略

## 6. 速率限制、配额与成本归集

### 5.1 速率限制

- 租户级：保护平台整体资源
- 用户级：防滥用与公平性
- 场景级：对高成本场景设置更严阈值

### 5.2 配额与预算

- 配额维度：租户/用户/空间/工具/模型
- 预算策略：阈值告警、硬上限拒绝、软上限降级到快模型

### 5.3 成本归集

- usage：token/请求数/耗时等
- 归因：关联到 runId/stepId/toolRef，便于运营与优化

## 7. 与 Safety/DLP 的协作

治理点：
- 输入侧：提示注入检测、敏感信息识别与脱敏
- 输出侧：敏感信息过滤、结构化约束、风险分级标记

输出结果建议包含：
- safetyFlags（命中规则摘要）
- policyHints（可选）：对下游动作的风险提示（不替代 AuthZ）

## 8. 审计与可回归

必须写审计的要点：
- routingDecision 摘要
- usage 与 latency
- 错误分类与降级链路触发原因
- inputDigest/outputDigest（脱敏）

回归要求：
- 模型/提示模板/路由策略变更必须绑定回放评测集
- 不通过评测门槛不发布；支持快速回滚

## 9. 可观测性

- 指标：成功率、错误率、P50/P95 延迟、熔断次数、降级触发率、配额命中率
- 成本：token 消耗分布、单租户/单工具成本、异常突增告警
- 追踪：traceId 串联到对话/工具/工作流

## 10. 演进路线

- MVP：多模型接入 + 基础路由 + 限流 + 审计
- V2：降级/熔断 + 配额/预算 + 成本归集
- V3：结构化输出强化 + 回放评测准入与自动回滚联动
